## Orphaned Events

Three event types are queried but never written:

1. **`onboarding_start`** ŌĆö Queried in `daily-metrics.ts:57` for the `onboardingStarts` metric. No code writes this event. The `DailyMetrics.onboardingStarts` column will always be 0.

2. **`chat_opened`** ŌĆö Queried in `daily-metrics.ts:92` for the `chatsOpened` metric and in `weekly-spark.ts:67` for spark eligibility filtering. No code writes this event. Chat happens in Telegram via deep link ŌĆö the app has no visibility into whether users actually open the Telegram chat.

3. **`verification_requested`** ŌĆö Queried in `admin.ts:219` to list pending verifications. No code writes this event. The `POST /api/verify/submit` route likely needs to create this event.

## Identity Management

No external analytics SDK is used. User identity is managed through:

- **Authentication:** Telegram `initData` HMAC-SHA256 validation ŌåÆ JWT with `userId` claim
- **Event attribution:** Every `Event` record has a `userId` foreign key pointing to the `User` table
- **External ID mapping:** `User.telegramId` (BigInt) maps to Telegram's user ID
- **No identify/group calls:** Not applicable ŌĆö no external analytics destination

## Observed Patterns

### Naming
- All event types use `snake_case` consistently
- Notification events use a dynamic prefix pattern: `notification:{type}` (e.g., `notification:daily_batch`)
- Verification events use separate names for approve/reject rather than a status property

### Architecture
- All 17 files call `prisma.event.create()` or `prisma.event.createMany()` directly ŌĆö no centralized tracking wrapper or helper function
- All tracking is server-side ŌĆö no client-side (frontend) events
- Event writes are `await`ed (blocking) without dedicated try/catch ŌĆö failures propagate to the route handler's catch block
- Batch events (intros, weekly-spark, intro-expiry) use `createMany` to write multiple records atomically

### Deduplication
- `re-engagement.ts` queries existing `notification:re_engagement` events before creating new ones (prevents double-sending at the same tier)
- `weekly-spark.ts` queries existing `weekly_spark` and `chat_opened` events for dedup and eligibility

### Double-write observation
- `premium_purchased` is written in two places: `premium.service.ts:85` (called by API when processing payment) and `bot/index.ts:186` (called by bot on `successful_payment` webhook). If both code paths execute for the same Telegram payment, two `premium_purchased` events would be created for the same transaction.

### Metrics rollup
- `DailyMetrics` cron (daily-metrics.ts) reads 12 event types to compute daily aggregates
- 2 of those 12 types are orphaned (`onboarding_start`, `chat_opened`), meaning those metrics columns are always 0

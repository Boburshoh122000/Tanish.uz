# Current Implementation — Tanish Tracking Architecture

**Last updated:** 2026-03-26

## SDK and Version

No external analytics SDK. Tracking uses an in-house PostgreSQL `Event` table via Prisma ORM.

```prisma
model Event {
  id        String   @id @default(cuid())
  userId    String
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  type      String
  metadata  Json?
  createdAt DateTime @default(now())

  @@index([type, createdAt])
  @@index([userId, createdAt])
}
```

Prisma version: `^6.19.2` (from pnpm lockfile). Client: `@prisma/client ^7.5.0`.

## Initialization

No SDK initialization step. Prisma client is instantiated once in `packages/api/src/index.ts` and imported by all route/service/cron modules. The bot package (`packages/bot/src/index.ts`) has its own Prisma client instance.

## Client vs Server

**Server-only.** All 32 event types are written server-side:
- 12 files in `packages/api/src/routes/`
- 3 files in `packages/api/src/services/`
- 4 files in `packages/api/src/cron/`
- 1 file in `packages/bot/src/`

The frontend (`packages/web/`) contains zero tracking calls. All user actions are tracked when the API processes the corresponding HTTP request.

## Call Routing

**Direct, scattered calls.** There is no centralized tracking wrapper, helper function, or event emitter. Each file calls `prisma.event.create()` or `prisma.event.createMany()` directly at the point of instrumentation.

Pattern:
```typescript
// Typical inline tracking call
await prisma.event.create({
  data: { userId, type: 'event_name', metadata: { key: value } },
});
```

For batch events (two users affected by one action):
```typescript
await prisma.event.createMany({
  data: [
    { userId: userA, type: 'event_name', metadata: { ... } },
    { userId: userB, type: 'event_name', metadata: { ... } },
  ],
});
```

## Identity Management

- **User ID:** CUID, generated by Prisma `@default(cuid())`
- **External ID:** `telegramId` (BigInt) — Telegram's native user identifier
- **Attribution:** Every Event record has a `userId` FK to the User table
- **Auth flow:** Telegram `initData` → HMAC-SHA256 validation → JWT with `{ userId, telegramId }` → userId extracted in auth middleware and attached to `request.userId`
- **No identify/group calls:** Not applicable — no external analytics destination

## Environment Variables

No analytics-specific environment variables. The Event table uses the same `DATABASE_URL` as all other Prisma models.

## Error Handling

Event writes are `await`ed inline (blocking). No dedicated try/catch wraps event creation — errors propagate to the outer route handler's try/catch block, which returns `{ success: false, error }`. This means a failed event write could cause the associated user action to appear to fail.

Exception: `creditReferral()` in `referrals.ts` wraps the entire function (including event writes) in a try/catch that returns `false` on failure — referral tracking failure is silent.

## Shutdown / Flush

No explicit shutdown or flush handling for analytics. Events are written synchronously to PostgreSQL as part of request handling — no in-memory buffer or queue for event data.

Notification delivery uses BullMQ (Redis-backed queue), but the `notification:*` event records are written to PostgreSQL after successful Telegram API delivery, not queued.

## Aggregation

`DailyMetrics` cron (`daily-metrics.ts`) runs at 02:00 UTC, queries the Event table for the previous day, and writes a single `DailyMetrics` record with 16 pre-computed columns. This is the only aggregation layer — no materialized views, no streaming aggregation.

Admin dashboard (`admin.ts`) also queries Event table directly for real-time today counts (DAU, matches, intros).

# Product: Tanish

**Last updated:** 2026-03-26
**Method:** codebase scan + conversation

## Product Identity
- **One-liner:** Young professionals in Tashkent open Tanish once a day to review a small curated batch of people, send an icebreaker intro to someone interesting, and — if both answer — get connected to chat in Telegram. The scarcity is the product: no swiping, no infinite scroll, just a daily ritual.
- **Category:** consumer social / people discovery
- **Product type:** B2C — no accounts, organizations, or multi-tenancy. Every entity is a single user.
- **Collaboration:** multiplayer (two-sided matching between users)

## Business Model
- **Monetization:** freemium via Telegram Stars (in-app currency)
- **Pricing tiers:**
  - **Free:** 3 daily matches, 5 daily intros
  - **Premium (150 Stars / ~$3/month):** 8 daily matches, 15 daily intros, +200 phantom ELO boost during batch generation
  - Promo price: 75 Stars (for launches / campaigns)
  - Grace period: 3 days after subscription lapses before downgrade
- **Billing integration:** Telegram Payments API (Stars). `pre_checkout_query` → `successful_payment` flow handled by Grammy bot. No Stripe, no external payment processor.

## Tech Stack
- **Primary language:** TypeScript (strict mode, shared across all packages)
- **Framework:** Fastify (API), React 18 + Vite (frontend SPA), Grammy (Telegram bot)
- **Database:** PostgreSQL via Prisma ORM (12 models, Railway-hosted)
- **Background jobs:** BullMQ (notification queue) + node-cron (batch generation, ELO decay, metrics rollup, intro expiry, re-engagement, premium expiry, weekly spark — 9 cron jobs total)
- **Cache/state:** Redis via ioredis (session cache, rate limiting, ELO sorted sets, daily likes counter, BullMQ backing store)
- **Photo storage:** Cloudflare R2 (S3-compatible)
- **Module organization:** pnpm monorepo with 6 packages: `shared` (types/schemas), `prisma` (schema/migrations), `api` (Fastify server), `bot` (Grammy), `web` (React SPA), `matching` (pure scoring algorithm)
- **Hosting:** Railway (API + Bot + PostgreSQL + Redis), Cloudflare Pages (frontend), Cloudflare R2 (photos)

## Value Mapping

### Primary Value Action
**Complete the daily cycle** — user opens Tanish, reviews their curated batch, and either sends an icebreaker or consciously passes. If this drops to zero, the product has failed. The value is in the constrained, daily ritual — not in time spent or volume of interactions.

### Core Features (directly deliver value)
1. **Daily batch generation** — the 5-factor matching engine (interest overlap 30%, profession match 25%, activity 20%, ELO proximity 15%, profile quality 10%) produces 3-8 curated profiles per user at 09:00 Tashkent time. This is the product's core promise: quality over quantity.
2. **Icebreaker intro system** — forced structured first contact via a shared-interest question. Prevents low-effort "hey" messages. 48-hour expiry creates urgency. The question-answer-answer flow is the "does the product work" mechanism.
3. **Match → Telegram chat handoff** — when both users answer, they get a deep link to each other's Telegram profile. Chat happens in Telegram. Tanish exits the interaction loop after creating the connection.

### Supporting Features (enable core actions)
1. **Onboarding (4 screens)** — collects gender, intent, city, birth date, role, university/workplace, interests, bio, languages, photos. Gates access to discovery (85% profile completeness threshold).
2. **ELO scoring** — invisible reputation system (200–2000 range, 3 tiers). Rewards engagement, penalizes inactivity and reports. Keeps the match pool healthy by deprioritizing low-quality profiles.
3. **Content filter** — strips URLs, phone numbers, @mentions from bios/intros. Flags profanity. Safety mechanism that protects the icebreaker experience.
4. **Report/block system** — 3 reports from different users = auto-suspension. Blocks are silent and mutual. Critical for user trust in a matching product.
5. **Premium (Telegram Stars)** — unlocks more daily matches and intros, ELO boost. Revenue mechanism + engagement lever.
6. **Notifications (BullMQ)** — daily batch ready, new intro received, match created, expiry warnings, re-engagement nudges (day 3/7/14/30), weekly spark. Quiet hours 23:00–08:00 Tashkent.
7. **Photo management** — upload to R2, position ordering, verification (manual admin review). Primary photo is critical for discovery card engagement.
8. **Referral system** — referral codes, ambassador flag, credit during onboarding. Built but not yet activated. Secondary to core loop validation.

## Entity Model

### Users
- **ID format:** CUID (e.g., `clx7abc123...`), generated by Prisma
- **External ID:** Telegram user ID (BigInt), stored as `telegramId`, used for auth and bot interactions
- **Roles:** regular user, premium user (time-limited subscription), admin (hardcoded `ADMIN_TELEGRAM_IDS` env var), ambassador (flag on user, not yet active)
- **Multi-account:** no — one Telegram account = one Tanish user, enforced by `telegramId` unique constraint
- **Statuses:** ACTIVE, SUSPENDED (auto after 3 reports), BANNED (manual admin action)

### No Group Hierarchy
This is a pure B2C product. There are no accounts, organizations, teams, or workspaces. The only entity is the individual user. City is a filter dimension on users, not a group entity.

## Current State
- **Existing tracking:** In-house PostgreSQL `Event` table with `type` (string) and `metadata` (JSON) fields. ~15 distinct event types written across routes, services, and cron jobs. `DailyMetrics` cron rolls up events into daily aggregates (DAU, signups, intros, matches, revenue, gender split, avg ELO).
- **Documentation:** no formal tracking plan. Event types are implicit from code.
- **Known gaps (owner-identified, priority order):**
  1. **In-session behavior black box** — no events between `app_open` and `intro_sent`. Cannot answer: did the user view their batch? How long did they spend on each card? Why didn't they send an intro?
  2. **Icebreaker quality → match correlation** — question text is stored on Intro records but not analyzed. No visibility into which questions drive higher `intro_answered` rates. Immediate actionable gap.
  3. **ELO calibration** — no feedback loop to validate whether ELO proximity correlates with match quality (both answer + chat link opened). The 5-factor scoring weights are speculative.
  4. **Retention cohort segmentation** — DailyMetrics computes D1/D7/D30 as single aggregate numbers. Cannot segment by acquisition channel, city, gender, or onboarding completeness. Hides divergent behavior (e.g., female 50% retention vs male 15% blended to 25%).
- **Kill metric:** gender ratio. If this is broken, nothing else matters for a matching product.

## Integration Targets
| Destination | Purpose | Priority |
|-------------|---------|----------|
| In-house (PostgreSQL Event table) | All event storage, metrics rollup, admin dashboard | Primary — current and near-term |
| PostHog (self-hosted) | Cohort analysis, retention curves, funnels by segment, A/B testing | Future — when DAU exceeds 5K |

**Decision rationale:** At target scale (500 signups, 150 DAU in 90 days), external tools are overkill. The Event table + DailyMetrics + admin dashboard (Phase 4) covers needs. PostHog self-hosted is the future choice for data sovereignty and zero per-event cost at scale. Segment/CDP is ruled out — single-product B2C app, no multi-tool routing needed.
